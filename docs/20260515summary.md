# 2026-05-15 工作总结

## 一、当前情况

当前仓库已经不再只是一个传统审计后台，而是在向“AI agent 审计工作台”转型，但这个转型还没有完全闭环。

从代码现状看：

- 前端主路径已经明显改造，核心页面不再只是管理台表格与统计卡片，而是围绕 `agent session / evidence / knowledge graph / report provenance` 来组织。
- 后端任务链已经具备“从文档到法规检索、风险评估、报告生成”的单次闭环能力，并且把阶段、事件、报告、发现项持久化给前端消费。
- 多 agent 当前仍然是“单次线性闭环”，不是“自我纠偏 loop”。也就是说，它可以完整走完一轮流程，但不会基于结果质量自动回退再查法规、再跑图谱、再重写报告。
- 知识图谱现在已经能作为证据界面参与使用，但还没有深入嵌入成 agent 推理中的强制回路。

当前工作树不是干净状态，存在大量历史未提交修改。根据当前 `git diff --stat`，工作树大约为：

- `80 files changed`
- `4725 insertions(+)`
- `2687 deletions(-)`

这说明当前仓库处于一个大规模演进中的中间状态，不适合轻率地按单个文件变化来做历史归因。

### 1. 前端主路径已切换为 agent workspace 风格

- `frontend/src/pages/DashboardPage.tsx`

当前前端的主要变化包括：

- 首页从“系统仪表盘”转向“agent control room”
- 文档页从“上传后结束”转向“上传后直接发起 agent session”
- 任务页从普通任务表切换为“agent workspace”，可展示进度、阶段、事件、文档状态、findings 和报告跳转
- 图谱页从运维式状态页改为可查询、可聚焦、可点击查看证据的知识图谱页面
- 报告页开始展示报告来源与模式，不再只是一个静态列表
- 登录、告警、设置页已经统一到新的产品风格中

### 2. 前后端 agent 闭环衔接已增强

接口和数据流方面，当前已经完成的关键改造包括：

- 前端 API 层补充 `getFindings` 等调用
- 前端类型中增加 `events / documents / report_id / report_metadata`
- 后端 `task_runner` 能区分单文档 agent 报告和聚合报告
- 报告 API 已返回 `report_metadata`
- 前端报告页已能展示 `report_source` 与 `report_mode`

这意味着前端已经不再只是被动等“任务完成”，而是开始具备展示 agent 执行过程和报告来源的能力。

### 3. 后端测试稳定性问题已部分收敛

此前 backend 集成测试中存在的 `database is locked` 问题，已经通过调整测试数据库生命周期和清理方式得到缓解。相关核心文件：

- `backend/tests/conftest.py`

当前后端关键 API 定向测试已可通过，不再在这组验证里复现锁库错误。

### 4. 报告生成链路的“来源不透明”问题已部分处理

此前系统存在多条报告生成路径：

- agent `report_writer` 自己写报告
- backend `task_runner` 聚合报告
- `reports/generate/{task_id}` 再调用 backend LLM 生成报告

现在已开始把“报告来源”显式暴露出来，前端也开始展示 provenance。虽然还没有从架构上完全收敛成唯一主路径，但用户可见性已经改善。

## 四、潜在问题

### 1. 当前仓库仍处于大工作树混合状态

虽然本轮已经完成多处收口，但当前仓库有大量历史未提交修改，包含：

- 旧改动
- 新增测试
- 前后端重构
- 配置与文档调整

潜在风险：

- 后续提交粒度难以保持清晰
- 很难用一次 commit 解释所有变化
- 若后续继续大改，回溯问题来源会越来越困难

### 2. agent 仍不是“真正形成 loop 的自纠偏系统”

当前 agent 的真实形态仍是：

- `parse_doc`
- `supervisor`
- `regulation_expert`
- `risk_assessor`
- `report_writer`
- `END`

潜在问题：

- 没有基于“法规不足”自动回退再查
- 没有基于“findings 为空”自动重试或重路由
- 没有基于“报告证据不充分”自动回到知识图谱或法规专家
- 没有真正的质量驱动循环

因此当前是“能走完一轮流程”，不是“能自我修正的 agent loop”。

### 3. 报告链路仍未完全收敛

虽然 provenance 已开始显式化，但目前仍存在多条报告路径：

- agent 直接产出
- task runner 聚合产出
- reports API 手动再生成

潜在问题：

- 用户看到的报告不一定是 agent 原始主报告
- 不同入口可能导致报告风格和内容差异
- 后续如果不收敛，维护成本会继续上升

### 4. 知识图谱仍偏“证据视图”，不是强约束推理环节

当前图谱页已经能用，但潜在问题仍包括：

- 主要还是人工证据查看界面
- agent 并未强制依赖图谱完成最终结论
- 图谱没有作为“缺证据时必须回查”的系统性机制

这意味着知识图谱已经可见，但还不是“agent 闭环的硬环节”。

### 5. 前端完整构建结果仍未形成有效结论

此前在当前环境中执行前端构建时，曾被环境侧错误阻断：

- `spawn EPERM`

潜在问题：

- 当前不能把“生产构建已通过”作为结论写死
- 当前通过的是 TypeScript 检查和定向测试，不等于完整生产打包完全无问题
- 问题可能来自本机子进程权限，也可能暴露出环境兼容性问题

### 6. 仓库中仍存在历史乱码与文案不一致问题

虽然本轮前端主页面已经基本清理，但仓库中仍然存在：

- 历史 summary 文档乱码
- 若干旧文件中残留中文编码问题
- 中英文文案风格不完全一致

潜在风险：

- 后续交接成本升高
- 文档可读性差
- 对外展示时体验不统一

### 7. backend 与 agent 的边界仍不够干净

当前桥接点：

- `backend/app/utils/agent_helpers.py`

仍有潜在架构问题：

- 使用 `sys.path` 注入方式桥接 agent
- backend 与 agent 没有形成真正稳定的包边界或服务边界
- 未来拆分部署或标准化测试时会继续成为隐患

建议下一步优先顺序如下：

1. 先补 `AuditTasks / KnowledgeGraph / Reports` 的页面级前端测试，扩大验证面

如果不按这个顺序推进，后续最容易出现的情况是：

- UI 看起来越来越像 agent

## 八、结论

截至 2026-05-15，当前项目已经从“传统审计后台”明显推进到“agent 审计工作台”的过渡阶段，前端体验、任务链路可见性、报告 provenance、部分 backend 测试稳定性都已经实质改善。

但项目仍然处在中间态：

- 真正的 agent loop 还没有建立
- 报告主链路还没有完全收敛
- 前端完整构建和更大范围测试还没完成
- 仓库工作树和历史文档质量仍有明显整理需求

因此，当前最准确的判断不是“已经完成”，而是：

**核心方向已经转正，关键链路已经打通一部分，但系统仍未进入可放心收口的稳定态。**
