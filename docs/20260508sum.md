## 2. 初始审查结论

在动手前，已确认的核心问题有：

### 2.1 启动与日志

- `backend/app/main.py` 里日志 handler 每次 startup 都重复注册。
- `RotatingFileHandler` 直接绑定 `data/logs/app.log`，测试或重复运行时容易触发 `PermissionError`。
- `data/logs/app.log` 中已经能看到重复日志输出，说明不是理论问题，而是实际发生。

### 2.2 认证与上传链路

- 原实现同时存在全局认证中间件和 `Depends(get_current_user)` 路由依赖。
- 这导致 multipart 上传路径被中间件先拦截，测试中实际出现 `401 Unauthorized`。
- 上传目录对当前环境并不总是可写，测试中实际出现 `PermissionError`。

### 2.3 Agent 状态机

- `agent/agents/supervisor.py` 原逻辑用 `matched_regulations` 和 `findings` 是否为空决定下一步。
- 这意味着：
  - 法规检索返回空列表时，supervisor 会反复回到 `regulation_expert`
  - 风险识别返回空 findings 时，supervisor 会反复回到 `risk_assessor`
- 即使系统没有异常，也可能出现“空结果无法收敛”的假死行为。

### 2.4 功能模块打通程度

- 前端主要接的是 `/api/audit/*`。
- 后端另有 `/api/agent-audit/*` 能力，但前端没有真实入口。
- 结果是“Agent 能力存在，但用户不能从页面直接使用”。

## 3. 已完成修改

### 3.1 后端启动与日志

已重写 `backend/app/main.py`，做了以下调整：

- 移除全局认证中间件，只保留路由级依赖认证。
- 新增 `_configure_logging()`，保证日志初始化幂等。
- 如果 `app.log` 无法打开，自动降级为仅控制台日志，不再让应用启动失败。
- 保留 JWT 默认密钥拦截逻辑。
- 根路径返回值改为正常英文 API 标识。

影响：

- 避免重复注册 handler。
- 避免测试/并发场景因日志文件占用而失败。
- 消除 multipart 上传在中间件层被错误拦截的问题。

### 3.2 文档上传与处理

已重写 `backend/app/api/documents.py`，关键变化：

- 上传逻辑保留安全文件名策略。
- 新增 `_get_upload_dir()`：
  - 优先使用 `settings.UPLOAD_DIR`
  - 若不可写，自动降级到系统临时目录 `tempfile.gettempdir()/gmpaudit_uploads`
- 统一了上传、批量上传、查询、处理、删除的错误处理。

影响：

- 解决测试环境和受限环境下 `data/documents` 不可写导致的失败。
- 上传功能不再依赖仓库目录一定可写。

### 3.3 Agent 状态机收敛修复

已修改：

- `agent/state.py`
- `agent/graph.py`
- `agent/agents/supervisor.py`
- `agent/agents/regulation_expert.py`
- `agent/agents/risk_assessor.py`
- `agent/agents/report_writer.py`

新增显式状态位：

- `regulation_checked`
- `risk_assessed`
- `report_generated`

具体变化：

- `parse_document_node` 初始化以上 3 个状态位为 `False`
- `regulation_expert` 完成后置 `regulation_checked=True`
- `risk_assessor` 完成后置 `risk_assessed=True`
- `report_writer` 完成后置 `report_generated=True`
- `supervisor` 改为基于这些布尔位进行路由，而不是依赖结果列表是否为空

影响：

- 即使法规结果为空，也能继续进入风险评估。
- 即使 findings 为空，也能继续生成报告并结束。
- workflow 从“数据驱动路由”改为“步骤完成驱动路由”，可收敛性显著提高。

### 3.4 审计任务 API 重构

已重写：

- `backend/app/api/audit.py`
- `backend/app/api/agent_audit.py`

主要改动：

- 统一初始化 agent state 的构造方式。
- 增加 task 与 agent 类型映射。
- 增加文档状态检查：只有 `processed` 文档可进入审计。
- `Finding` 落库时统一做类型和严重级别映射。
- 审计失败时统一写回：
  - `task.status = failed`
  - `task.error_message`
  - `task.completed_at`
- 审计成功时统一写回：
  - `task.status = completed`
  - `task.progress = 100`
  - `task.completed_at`
- `agent_audit` 后台任务也补齐 `document_id` 落库。

影响：

- `/api/audit/*` 与 `/api/agent-audit/*` 的行为更一致。
- 任务生命周期更完整，状态查询更有意义。
- 后续做前端轮询或告警联动更容易。

## 7. 关键设计决策记录

### 7.1 为什么移除全局认证中间件

原因：

- 路由依赖认证已经足够
- 中间件会和 multipart/form-data、测试 override、异常处理形成重复耦合
- 真实问题已在测试中暴露为 401

结论：

- 用单一认证机制比“双层认证”更稳

### 7.2 为什么状态机要引入显式布尔位

原因：

- “结果为空”不等于“步骤未执行”
- 监管检索为空、风险识别为空，在业务上都可能是合法结果
- 如果继续用空列表驱动路由，流程无法区分“没跑”和“跑完但为空”

结论：

- 要把“步骤是否完成”和“步骤产出内容”解耦

### 7.3 为什么上传目录要自动降级

原因：

- 当前环境下仓库目录并不总是可写
- 测试已实际证明会触发 `PermissionError`
- 上传链路应优先保证服务可用，而不是强依赖固定目录

结论：

- 保留首选目录
- 不可写时自动 fallback 到临时目录
